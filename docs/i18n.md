# Internationalization

Lifedeck is multilingual from the start. The user's language is detected from the
browser and falls back to English.

## How detection works

1. On each request, the root layout reads the `Accept-Language` header.
2. `@lifedeck/i18n`'s `detectLocale` parses the header, ranks entries by quality,
   and returns the first **supported** locale (matching on the base subtag, so
   `pt-BR` resolves to `pt`).
3. If nothing matches, it returns `DEFAULT_LOCALE` (`en`).
4. The resolved locale and its message catalog are passed to the client through
   `MessagesProvider`; components read them with `useI18n`.

## Supported locales

Declared in `packages/i18n/src/locales.ts`:

```ts
export const SUPPORTED_LOCALES = ['en', 'pt', 'es'] as const
export const DEFAULT_LOCALE = 'en'
```

English, Portuguese, and Spanish are all first-class. `en` is the canonical
catalog; its shape (`Messages`) is the contract every other locale must satisfy,
enforced at compile time. Localization reaches the whole product, not just the UI:

- **UI copy** - `src/messages/<code>.ts` (one catalog per locale).
- **Legal pages** - the Terms of Use and Privacy Policy bodies are structured
  data in `src/messages/legal-<code>.ts`, rendered through the i18n catalog
  (`messages.legal.terms` / `messages.legal.privacy`).
- **Emails** - `renderEmail` localizes verification, invitation, assignment, and
  digest templates; `toEmailLocale` maps a user's locale to the email locale.
- **AI generation** - the `GenerationBrief.locale` enum drives the language of the
  generated checklist.

## Adding a locale

1. Add the code (and its `Accept-Language` tags) to `SUPPORTED_LOCALES` /
   `LOCALE_TAGS` in `src/locales.ts`.
2. Create `src/messages/<code>.ts` exporting a `Messages` object, plus
   `src/messages/legal-<code>.ts` for the legal documents (TypeScript will flag
   any missing keys).
3. Register it in `messages` inside `src/index.ts`.
4. Add a detection test case, and an email-copy entry per template in
   `infrastructure`'s `render-email.ts` if the locale should localize emails.

## Adding a message

1. Add the key to the `Messages` type in `src/messages/types.ts`.
2. Provide the value in **every** locale catalog (compilation fails otherwise).
3. Read it via `useI18n().messages`.

## Design choices

Locale follows the browser: the server resolves it from `Accept-Language`
(SSR-friendly, no hydration flash) and the client forwards the browser language
as `Accept-Language` on every API call, so the backend localizes its own output
too. Locale-prefixed URLs and a manual language switcher are deliberately out of
scope - the typed catalogs and `Accept-Language` resolution already satisfy the
"browser default with English fallback" requirement without a routing rewrite.
